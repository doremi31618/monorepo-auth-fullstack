# 📜 Platform Charter – Foundation Guardrails

## 0. 文件定位（請先讀這段）

這份文件不是：

- PRD
- 技術設計文件
- Roadmap

這是一份 Decision Guardrail Document，用來：

> 保護這個平台不被短期需求、過度泛化、或錯誤抽象拖垮。

當出現任何「要不要把某個能力放進平台」的討論時，
這份文件的結論優先於任何直覺與臨時判斷。

---

## 1. 平台存在目的（Why）

本平台的存在目的不是實現某一個特定產品，而是：

> 集中並治理「所有產品都一定會需要、
> 但如果每次重做都會失控的基礎能力」，
> 讓未來的產品可以專注在自身的價值創造。

這個平台必須能夠作為以下產品的共同前身（但不綁定於任何一種）：

- 教育平台與個人化學習體驗
- 電商 / 購物網站
- 線上工具（SaaS / Utility）
- 內容平台 / 部落格
- 研究資料與實驗平台
- 其他尚未被定義的產品

---

## 2. 平台【一定負責】的事情（Platform Responsibilities）

以下責任被視為平台的一等公民（First-class responsibilities）。
任何產品模組 不得自行實作、繞過或複製 這些能力。

### 2.1 Identity & Access

- 使用者身份（Authentication / Session lifecycle / SSO）
- 權限與存取控制（RBAC + Policy Hook）
- 多情境存取（裝置 / 角色 / Scope）

### 2.2 Content & Asset Abstraction

- 內容的存在形式（不包含內容價值判斷）
- 檔案 / 資產管理（upload、metadata、ownership、reference）
- 資產與內容的關聯關係

### 2.3 Event & Behavior Capture

- 所有有意義的系統行為必須能被記錄為事件
- Event 是一等公民，而非 side-effect
- 支援後續分析、推薦、AI、稽核等用途

### 2.4 Auditability & Traceability

- 可追溯的系統操作紀錄
- 管理操作、資料存取、狀態變更
- 支援合規、研究、回溯與除錯

### 2.5 Observability & System Health

- 系統存活狀態（health / readiness）
- 系統行為可觀測性（latency / error / backlog）
- 跨模組行為追蹤能力（correlation / trace）

### 2.6 Governance & Evolution

- 模組邊界與依賴規則
- Schema ownership 與演進策略
- 版本、migration、rollback 規範
- 確保平台可以長期演進而不崩解

---

## 3. 平台【永遠不負責】的事情（Non-Responsibilities）

以下事項明確不屬於平台責任範圍。
即使它們看起來「很通用」、「之後一定會用到」，也不得進入平台。

### 3.1 業務邏輯與產品語意

- 不定義「學會 / 完成 / 成交 / 成功」的標準
- 不包含任何產品專屬邏輯
- 不包含教育、商業、研究的 domain decision

### 3.2 UI / UX 決策

- 不負責畫面設計
- 不定義使用流程
- 不提供產品級互動體驗

### 3.3 商業模式與金流

- 不處理訂閱、定價、折扣
- 不處理付款流程或金流邏輯

### 3.4 AI / 推薦 / 策略本身

- 不定義 prompt
- 不選擇模型
- 不實作推薦或評分演算法

平台只提供資料與事件管道。

### 3.5 內容價值與品質判斷

- 不評價內容好壞
- 不評價教材難度
- 不評價研究結果品質

平台是容器，不是裁判。

---

## 4. Decision Filter（平台決策過濾器）

任何新能力要進入平台層，必須同時滿足以下條件：

1. 跨至少兩個以上產品類型
2. 與業務語意無關
3. 不影響產品 UI / UX 決策
4. 不會限制未來演進方向

只要有一項不符合，該能力必須屬於產品模組。

---

## 5. Foundation Done 的定義（何時可以停）

平台 foundation 被視為「完成」，當且僅當：

- 所有平台責任（第 2 節）皆有穩定實作或明確接口
- 所有非責任事項（第 3 節）未被侵入平台
- 新產品可以在 不修改平台核心 的前提下開始開發
- 平台的下一步工作只剩「支援產品」，而非「打造平台本身」

> 一旦達成以上條件，
> 任何繼續擴張平台 scope 的行為都必須被視為風險。
